# Multi-tenant architecture

This document is the **pinned contract** between the conductor and the 4 task agents. Sub-agents porting runtime code from the internal pattern MUST follow this contract verbatim. Operators building custom task agents MUST conform to the same contract.

**Availability status.** This architecture is available now: the per-tenant conductor and the mail/calendar/files agents ship and are wired by setup, with Microsoft 365 as the supported provider today. Google (Gmail / Calendar / Drive) is a preview and is not yet functional (no Google MCP server ships in this release). The default conductor relay is the built-in browser/localhost-HTTP relay (binds to `127.0.0.1` only); the Discord / Slack / WhatsApp relay drivers described below are roadmap and not yet available. The voice and voice-call modules are a preview in this release.

## Per-tenant directory layout

```
$INSTALL_PATH/
  <slug>/                      # tenant config + state. Owned by service account <slug>-svc.
    .env                       # operator-edited config (chmod 600)
    .claude/settings.json      # MCP server config for task agents
    store/
      jobs.db                  # per-tenant SQLite job queue (see schema below)
      google-auth/             # Google OAuth token cache (if mail-google installed)
      ms365-auth/              # MS365 OAuth token cache (if mail-ms365 installed)
      whatsapp-bridge/         # WhatsApp Web session (if RELAY_TYPE=whatsapp)
      conversations.db         # conductor chat memory
    logs/
      audit.log                # per-tenant audit log

  <slug>-conductor/            # conductor runtime
    pbox-conductor.mjs         # the daemon
    package.json + node_modules/
    .claude/settings.json -> ../<slug>/.claude/settings.json (symlink)

  <slug>-mail/                 # mail task-agent runtime
    pbox-mail-agent.mjs
    package.json + node_modules/

  <slug>-calendar/             # calendar task-agent runtime
    pbox-calendar-agent.mjs
    package.json + node_modules/

  <slug>-files/                # files task-agent runtime
    pbox-files-agent.mjs
    package.json + node_modules/

  <slug>-voice/                # voice task-agent runtime (TTS + STT)
    pbox-voice-agent.mjs
    package.json + node_modules/

  <slug>-voice-call/           # real-time voice call orchestrator (Gemini Live)
    pbox-voice-call.mjs
    public/                    # browser-default UI assets
    package.json + node_modules/
```

## LaunchDaemons per tenant

```
${LAUNCHDAEMON_PREFIX}.<slug>-conductor
${LAUNCHDAEMON_PREFIX}.<slug>-mail        (loaded iff GOOGLE_CLIENT_ID or MS365_CLIENT_ID set)
${LAUNCHDAEMON_PREFIX}.<slug>-calendar    (loaded iff CALENDAR_ENABLED=true)
${LAUNCHDAEMON_PREFIX}.<slug>-files       (loaded iff FILES_ENABLED=true)
${LAUNCHDAEMON_PREFIX}.<slug>-voice       (loaded iff ELEVENLABS_API_KEY or GROQ_API_KEY set)
${LAUNCHDAEMON_PREFIX}.<slug>-voice-call  (loaded iff GOOGLE_API_KEY set; persistent HTTP+WSS service, not a poller)
```

The `voice-call` daemon is operationally distinct: persistent service on `127.0.0.1:${VOICE_CALL_PORT}/`, bridges browser audio capture to Gemini Live's bidirectional audio API. Voice and voice-call are a preview in this release (unverified, not certified). They are conversation-only; in-call tool dispatch is planned for a future release alongside a synchronous conductor API.

## Job queue schema (`<slug>/store/jobs.db`)

```sql
CREATE TABLE jobs (
  id              TEXT PRIMARY KEY,        -- crypto.randomUUID()
  task_type       TEXT NOT NULL
                  CHECK (task_type IN ('mail', 'calendar', 'files', 'voice')),
  prompt          TEXT NOT NULL,           -- operator-or-relay-provided text
  status          TEXT NOT NULL DEFAULT 'PENDING_REVIEW'
                  CHECK (status IN (
                    'PENDING_REVIEW','APPROVED','REJECTED','IN_PROGRESS',
                    'COMPLETED','FAILED','BLOCKED'
                  )),
  risk_level      TEXT NOT NULL DEFAULT 'standard'
                  CHECK (risk_level IN ('standard', 'high')),
  conductor_ref   TEXT,                    -- e.g. discord-msg-id, slack-ts, web-session-id
  result          TEXT,                    -- task agent's final response
  reviewer_note   TEXT,                    -- if APPROVED/REJECTED came from content-classifier or operator review
  cost_usd        REAL,
  created_at      INTEGER NOT NULL,        -- Date.now()
  updated_at      INTEGER NOT NULL,
  completed_at    INTEGER,
  last_active     INTEGER                  -- heartbeat: task agent writes Date.now() every HEARTBEAT_MS
);

CREATE TABLE job_events (
  id          TEXT PRIMARY KEY,
  job_id      TEXT NOT NULL,
  event_type  TEXT NOT NULL,               -- 'created'|'reviewed'|'started'|'tool_call'|'completed'|'failed'
  actor       TEXT NOT NULL,               -- 'conductor'|'mail-agent'|'content-classifier'|'operator'
  detail      TEXT,                        -- JSON
  created_at  INTEGER NOT NULL
);
```

**Status workflow:**
```
PENDING_REVIEW            <- conductor inserts a new job
   │
   ├── content-classifier auto-classifies risk:
   │     standard -> APPROVED        (or operator-confirm-required)
   │     high     -> awaits operator approval
   │
   ▼
APPROVED                  <- task agent picks up
   │
   ▼
IN_PROGRESS               <- heartbeat starts
   │
   ▼
COMPLETED / FAILED / BLOCKED
```

If `content-classifier` module is NOT installed, jobs go straight `PENDING_REVIEW → APPROVED` (or `PENDING_REVIEW → IN_PROGRESS` with no review step) — single-operator deployments may not need the review layer.

## Conductor responsibilities

1. Load relay driver based on `RELAY_TYPE`. The default is the built-in browser/localhost-HTTP relay (works out of the box, `127.0.0.1` only); the Discord / Slack / WhatsApp drivers are roadmap and not yet available. Dispatch incoming messages to the right task type. Persist conductor_ref for later reply-back routing.
2. Insert jobs into `<slug>/store/jobs.db` with `status='PENDING_REVIEW'`.
3. Poll for COMPLETED jobs, route the result back via the relay driver.
4. Maintain conversation memory in `<slug>/store/conversations.db`.
5. NEVER hold credentials directly. Task agents read their own provider creds from `<slug>/.env`.

## Task-agent responsibilities

1. Poll `<slug>/store/jobs.db` for APPROVED jobs matching the agent's `task_type`. Default poll interval: 30s.
2. Heartbeat: write `Date.now()` to `last_active` every 15s while a job is in IN_PROGRESS.
3. Use `@anthropic-ai/claude-agent-sdk` (the `query` function) to execute the task.
4. Load MCP servers from `<slug>/.claude/settings.json` (the agent's MCP config is per-tenant).
5. Enforce tool-call retry caps (default 10 calls per tool per job).
6. Apply provider-specific guards: domain allowlist for email-send (operator-edited `<slug>/store/email-allowlist.txt`); secret-scan on outbound text.
7. Write audit-log entries to `<slug>/logs/audit.log`.
8. Update job status to COMPLETED with result on success, FAILED with error on exception.

## What is OUT of scope for the public runtime port

- The Telegram-bot relay (`grammy` + bot-token) found in the reference implementation. The public release default is the built-in browser/localhost-HTTP relay; Discord / Slack / WhatsApp drivers are roadmap and not yet available.
- The internal diagnostic broker (port 8444). Internal-only tool.
- Internal approval-queue daemon. Public release uses the content-classifier sidecar.
- Internal shared adapter imports (`shared/anthropic-claude-adapter.mjs`). Use `@anthropic-ai/claude-agent-sdk` directly.
- Internal global job-queue path. Public jobs DB is per-tenant under `<slug>/store/`.
- Internal global audit-log path. Public audit is per-tenant under `<slug>/logs/`.
- Hardcoded tenant lists (allowlist domain enums, company enums, etc). Public release reads these from operator-edited per-tenant files such as `<slug>/store/email-allowlist.txt`.
- Internal-only modules: workshop dashboard, RBAC overlay, diagnostic broker handlers.

## Sanitisation policy for runtime ports

Sub-agents porting code MUST run `bash hooks/pre-push` from the repo root before reporting done. The sanitisation patterns at `~/.config/pandoras-box/sanitize-patterns` are the authoritative banned-token list (NOT duplicated here, by design — the patterns evolve and the gate is the source of truth).

The canonical public-release substitutions:
- Internal tenant identifiers → `$COMPANY_SLUG` env var
- Internal absolute paths → `${INSTALL_PATH}/<slug>/...` templated from env
- Internal LaunchDaemon label prefix → `${LAUNCHDAEMON_PREFIX}` env var (theme.conf-set)
- Internal email-domain enums → operator-editable `<slug>/store/email-allowlist.txt`
- Internal MCP server refs → operator-editable `<slug>/.claude/settings.json`

Any port that lands a banned token will be blocked by the pre-push gate. The sanitisation infrastructure is what catches drift; this document doesn't try to enumerate what the gate already enforces.

Public release uses tenant-agnostic equivalents via env interpolation: `$COMPANY_SLUG`, `$INSTALL_PATH/<slug>/`, `com.pandoras-box.` (WITH hyphen).
